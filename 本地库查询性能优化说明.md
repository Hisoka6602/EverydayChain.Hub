# 本地库查询性能优化说明

## 1. 高频 API 查询路径
- 业务任务查询：`api/v1/business-query/tasks`
- 异常件查询：`api/v1/business-query/exceptions`
- 回流记录查询：`api/v1/business-query/recirculations`
- 总看板：`api/v1/dashboard/overview`
- 码头看板：`api/v1/dock-dashboard/overview`
- 分拣报表：`api/v1/reports/query`、`api/v1/reports/export/csv`

## 2. 优化前最大瓶颈
- 跨分表页码分页依赖 `skip/take`，且存在跨分表重复计数。
- 看板/报表高频请求每次实时聚合，数据库与应用层合并压力高。
- `DbContext` 工厂未池化，查询高并发场景存在上下文创建开销。
- 高频筛选存在 `Trim()` 与 OR 主路径，影响索引命中与计划稳定性。

## 3. 本次优化内容
- 业务任务查询新增游标分页主路径，并保留页码分页兼容。
- 跨分表页码分页改为单次遍历返回总数与分页结果，减少重复计数往返。
- 增加分表 legacy 空后缀兜底开关，支持无历史表环境关闭兜底。
- 总看板/码头看板/报表查询增加可配置短 TTL 本地缓存。
- `AddDbContextFactory` 切换为 `AddPooledDbContextFactory`，池大小可配置。
- 新增 `NormalizedWaveCode`、`NormalizedBarcode`、`ResolvedDockCode` 字段并用于高频查询主路径。

## 4. 已切换为游标分页的查询
- 业务任务查询
- 异常件查询
- 回流记录查询

说明：当请求同时传入 `LastCreatedTimeLocal` 与 `LastId` 时自动走游标分页；否则保持页码分页。

## 5. 已增加短 TTL 缓存的查询
- `GlobalDashboardQueryService`（`QueryCache.GlobalDashboardSeconds`）
- `DockDashboardQueryService`（`QueryCache.DockDashboardSeconds`）
- `SortingReportQueryService`（`QueryCache.SortingReportSeconds`）

说明：仅缓存查询结果，不缓存 `DbContext`；实时链路（扫描上传、请求格口、落格回传、WMS 回写）未加缓存。

## 6. 索引命中写法调整
- 波次筛选改为 `NormalizedWaveCode` 等值匹配。
- 条码筛选改为 `NormalizedBarcode` 等值匹配。
- 码头/格口筛选改为 `ResolvedDockCode` 主路径。
- 去除 `Trim()` 作为主过滤条件，去除 `ActualChuteCode/TargetChuteCode` OR 主路径。

## 7. DbContext 池化配置
- 配置节点：

```jsonc
"EfCore": {
  "DbContextPoolSize": 256
}
```

- 生效位置：`ServiceCollectionExtensions.AddInfrastructure`。

## 8. 索引复核与新增
- 已保留并继续使用：`(CreatedTimeLocal, Id)`（游标排序键）。
- 已新增：
  - `NormalizedBarcode`
  - `NormalizedWaveCode`
  - `(NormalizedWaveCode, CreatedTimeLocal)`
  - `ResolvedDockCode`
  - `(ResolvedDockCode, CreatedTimeLocal)`
  - `(CreatedTimeLocal, NormalizedWaveCode, ResolvedDockCode)`
- 已保留并复核：
  - `(FeedbackStatus, CreatedTimeLocal)`
  - `(FeedbackStatus, IsFeedbackReported, FeedbackTimeLocal)`

## 9. 旧实现替换清单
- 已替换：跨分表分页“先全量 `CountAsync` 再分页查询”的双遍历路径。
- 已替换：`Trim()` 与 OR 作为高频 SQL 主过滤路径。
- 已替换：看板/报表每次全量实时聚合主路径（改为可配置短 TTL 缓存）。
- 已替换：无条件附带 legacy 空后缀表读取（改为配置开关）。

## 10. 当前剩余性能边界
- 页码分页在超大页码下仍存在 `skip` 天然开销，建议高并发调用统一使用游标分页。
- 跨分表聚合仍为“分表聚合结果合并”模式，分表数量极大时存在线性增长边界。
- 报表导出 CSV 受结果集规模影响，建议结合时间窗口与码头筛选控制数据量。

## 11. 上线后建议观察指标
- API：P95/P99 延迟、QPS、5xx 比例。
- 数据库：CPU、逻辑读、锁等待、慢 SQL 数量。
- 应用：GC 次数与暂停时间、线程池排队、内存占用。
- 缓存：命中率、逐接口缓存键数量、TTL 过期节奏。
- 分页：游标模式占比、大页码访问占比（用于持续压降 `skip` 路径）。
