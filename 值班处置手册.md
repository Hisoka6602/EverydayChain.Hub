# EverydayChain.Hub 值班处置手册

## 说明

本手册适用于 EverydayChain.Hub 同步服务的**日常值班与故障应急处置**。  
每次告警触发后，值班人员应按照对应章节的步骤逐项执行，并在处置记录模板中留档。

---

## 一、告警优先级定义

| 优先级 | 描述 | 响应时间目标 |
|--------|------|-------------|
| P0（致命） | 服务进程停止、检查点损坏、数据写入全量失败 | 15 分钟内响应 |
| P1（严重） | 全部或大多数表同步持续失败、磁盘空间不足（低于 200 MB） | 30 分钟内响应 |
| P2（警告） | 部分表同步失败、内存水位持续告警、熔断器打开、卡死检测触发 | 2 小时内响应 |
| P3（提示） | 单次超时、归档空间超过 1 GB、日志无活跃写入警告 | 下一工作日处理 |

---

## 二、告警处置步骤

### 2.1 同步后台任务疑似卡死（WatchdogTimeoutSeconds 触发）

**日志关键字**：`同步后台任务疑似卡死` / `Critical`  
**优先级**：P2

**处置步骤**：
1. 检查服务进程是否仍在运行：
   ```bash
   pgrep -f EverydayChain.Hub.Host
   ```
2. 检查最近日志是否有新输出（观察 logs/ 目录最新日志文件）：
   ```bash
   ls -lt logs/*.log | head -5
   tail -50 logs/sync-latest.log
   ```
3. 若进程存活但日志无新输出超过 30 分钟：
   ```bash
   # 执行一键体检确认状态
   bash scripts/health-check.sh
   ```
4. 若体检发现磁盘满或权限问题，先处理环境问题再重启。
5. 若环境正常，重启服务：
   ```bash
   # 根据实际部署方式选择其中一种
   systemctl restart everydaychain-hub     # systemd 部署
   docker restart everydaychain-hub        # Docker 部署
   dotnet EverydayChain.Hub.Host.dll &     # 直接启动
   ```
6. 重启后观察日志 5 分钟，确认同步正常推进。

---

### 2.2 磁盘可用空间不足

**日志关键字**：`磁盘可用空间不足` / `启动自检失败` / `关键写入前磁盘检查失败`  
**优先级**：P1（< 200 MB）/ P2（< 500 MB）

**处置步骤**：
1. 查看当前磁盘使用情况：
   ```bash
   df -h
   du -sh data/ logs/
   ```
2. 检查日志归档是否过多：
   ```bash
   ls -lh logs/*.log* | head -20
   # 手动清理 60 天以前的日志文件
   find logs/ -name "*.log*" -mtime +60 -delete
   ```
3. 检查数据目录压缩归档：
   ```bash
   ls -lh data/*.json.gz | wc -l
   # 若归档过多，执行清理（保留 7 个）
   bash scripts/disaster-recovery.sh archive-cleanup
   ```
4. 若磁盘仍然不足，暂时关闭归档功能（热更新无效，需修改 appsettings.json 后重启）：
   ```json
   "TargetStoreArchiveMaxCount": 0
   ```
5. 确保剩余可用空间至少 500 MB 后重启服务。

---

### 2.3 单表内存水位告警

**日志关键字**：`单表内存水位告警`  
**优先级**：P2

**处置步骤**：
1. 查看告警日志确认哪个表触发，以及当前内存大小。
2. 检查该表是否长期未被驱逐（观察 `EvictIdleTablesAsync` 驱逐日志）。
3. 短期应对：确认空闲驱逐已开启（`EnableIdleEviction=true`）并降低阈值：
   ```json
   "IdleEvictionThresholdMinutes": 15
   ```
4. 若内存持续膨胀无法收敛，考虑缩短该表的 `PollingIntervalSeconds`，减少单次加载的数据量。
5. 若 `TableMemoryWarningThresholdMb` 设置过低导致频繁误报，可适当上调：
   ```json
   "TableMemoryWarningThresholdMb": 512
   ```

---

### 2.4 同步整轮执行超时

**日志关键字**：`同步整轮执行超时`  
**优先级**：P2

**处置步骤**：
1. 确认当前 `TableSyncTimeoutSeconds` 配置值（默认 600 秒）。
2. 查看超时前最后一次表级日志，定位耗时最长的表。
3. 若大表正常，可上调超时阈值（修改 appsettings.json 后重启）：
   ```json
   "TableSyncTimeoutSeconds": 1200
   ```
4. 若某表耗时异常（超出预期数倍），检查 Oracle 侧慢查询日志，评估是否需要添加索引。
5. 若轮询间隔小于超时时间，应确保 `PollingIntervalSeconds` > `TableSyncTimeoutSeconds`，避免轮次重叠。

---

### 2.5 熔断器打开

**日志关键字**：`熔断器打开` / `CircuitBreaker`  
**优先级**：P2

**处置步骤**：
1. 熔断器打开说明 Oracle 或 SQL Server 连续失败率超过阈值，首先排查网络与数据库状态：
   ```bash
   # 检查 Oracle 连通性（telnet 或 nc）
   nc -zvw5 <oracle-host> 1521
   # 检查 SQL Server 连通性
   nc -zvw5 <sqlserver-host> 1433
   ```
2. 若网络正常，检查数据库账号密码是否未过期。
3. 熔断器会在 `CircuitBreakerBreakDurationSeconds`（默认 20 秒）后自动尝试半开状态，依赖恢复后自动关闭，**无需手动干预**。
4. 若熔断器持续打开超过 5 分钟，检查 Oracle 服务状态并联系 DBA。
5. 若需调整熔断策略，修改 appsettings.json 后重启：
   ```json
   "CircuitBreakerBreakDurationSeconds": 60,
   "CircuitBreakerFailureRatio": 0.8
   ```

---

### 2.6 检查点文件损坏/丢失

**日志关键字**：`读取检查点文件失败` / `JSON 格式非法`  
**优先级**：P0（若导致服务启动失败）

**处置步骤**：
1. 停止服务（避免写入覆盖备份）。
2. 查看是否有自动备份文件（`.bak.*` 后缀）：
   ```bash
   ls -lt data/sync-checkpoints.json*
   ```
3. 若有可用备份，手动恢复：
   ```bash
   cp data/sync-checkpoints.json.bak.<timestamp> data/sync-checkpoints.json
   ```
4. 若无备份，执行检查点重置（服务将从 StartTimeLocal 重新全量同步）：
   ```bash
   bash scripts/disaster-recovery.sh checkpoint-reset
   ```
5. 重启服务，观察日志确认同步从预期起始时间推进。

---

### 2.7 目标快照文件损坏/丢失

**日志关键字**：`读取同步目标落地文件失败` / `反序列化失败`  
**优先级**：P1

**处置步骤**：
1. 停止服务。
2. 确认损坏的表编码（从错误日志中提取 `TableCode`）。
3. 尝试从压缩归档恢复：
   ```bash
   bash scripts/disaster-recovery.sh snapshot-restore --table-code <TableCode>
   ```
4. 若无归档，直接删除损坏的快照文件（服务重启后将重新从 Oracle 全量读取）：
   ```bash
   rm data/sync-target-store.<TableCode>.json
   ```
5. 同步执行检查点重置（或仅清空该表的检查点游标）后重启服务。
6. 观察日志，确认该表从历史时间点重新同步直至追平当前。

---

### 2.8 压缩归档写入失败

**日志关键字**：`压缩归档旧版本目标快照文件失败`  
**优先级**：P3

**处置步骤**：
1. 通常由磁盘空间不足或权限问题引起，先执行体检：
   ```bash
   bash scripts/health-check.sh
   ```
2. 清理旧日志和归档后，检查是否有残留 `.bak` 文件：
   ```bash
   ls -lh data/*.bak*
   # 若有残留 .bak 文件，手动删除或手动归档
   rm data/sync-target-store.*.bak*
   ```
3. 服务无需重启，下次写入时会重试归档操作。

---

### 2.9 服务进程停止/未运行

**日志关键字**：`health-check.sh` 进程存活检查 `[FAIL]`  
**优先级**：P0

**处置步骤**：
1. 查看系统日志确认退出原因：
   ```bash
   journalctl -u everydaychain-hub --since "1 hour ago"  # systemd
   docker logs everydaychain-hub --tail 100              # Docker
   ```
2. 重启前先执行一键体检，确认环境正常：
   ```bash
   bash scripts/health-check.sh
   ```
3. 修复环境问题后重启服务。
4. 若服务频繁崩溃（24 小时内超过 3 次），按 P0 升级处理，并保留现场日志用于根因分析。

## 三、常见告警快速参考

| 告警关键字 | 优先级 | 可能原因 | 处理章节 |
|---|---|---|---|
| `同步后台任务疑似卡死` | P2 | 主循环阻塞/死锁 | 2.1 |
| `磁盘可用空间不足` | P1/P2 | 日志/归档积压 | 2.2 |
| `单表内存水位告警` | P2 | 大表内存膨胀 | 2.3 |
| `同步整轮执行超时` | P2 | 大表耗时超限 | 2.4 |
| `熔断器打开` | P2 | Oracle/SQL Server 连续失败 | 2.5 |
| `读取检查点文件失败` | P0 | 检查点损坏或丢失 | 2.6 |
| `读取同步目标落地文件失败` | P1 | 快照损坏或丢失 | 2.7 |
| `压缩归档旧版本目标快照文件失败` | P3 | 磁盘/权限问题 | 2.8 |
| 进程存活检查 `[FAIL]` | P0 | 服务进程停止 | 2.9 |
| `本轮同步存在失败表` | P2 | 单表或多表同步失败 | 2.4/2.5 |

## 四、升级规则

| 场景 | 升级操作 |
|------|---------|
| P0 告警处置后 15 分钟仍未恢复 | 升级至技术负责人，发送服务中断通知 |
| P1 告警响应后 1 小时内仍未恢复 | 升级至技术负责人 |
| 数据滞后超过 2 倍 `DefaultMaxLagMinutes` | 通知业务方评估影响 |
| 24 小时内同一告警出现 3 次以上 | 提交改进工单，纳入下一迭代优化 |
