# 当前程序能力、功能与缺陷分析

## 1. 当前程序能力概览

基于当前代码，项目已具备以下核心能力：

1. **Oracle→SQLServer 的增量同步编排能力**  
   可按表配置加载任务、读取检查点、计算窗口、并发执行多表同步，并支持按优先级排序执行。

2. **分页读取 + 暂存 + 幂等合并能力**  
   同步链路已具备分页读取源数据、写入暂存、按业务键进行插入/更新/跳过的幂等合并流程。

3. **删除识别与删除执行能力（含 dry-run）**  
   可在窗口内对比源端与目标端业务键差异，执行软删/硬删/禁用删除策略，并输出删除审计。

4. **检查点持久化与失败恢复能力**  
   通过文件检查点保存每表最近成功游标、批次信息和错误信息，支持断点续跑。

5. **分表保留期治理能力**  
   项目中已具备分表识别、保留期清理、dry-run、审计与回滚脚本生成能力（由保留期执行链路承载）。

6. **基础可观测能力**  
   同步执行输出 Lag/Backlog/吞吐/失败率等指标日志，具备基础运行态观测数据。

## 2. 已实现功能清单（按执行链路）

### 2.1 同步编排层
- 读取启用表配置并按优先级排序。
- 支持多表并发执行（受 MaxParallelTables 控制）。
- 每次执行按检查点计算本地时间窗口。

### 2.2 同步执行层
- 批次创建、状态流转（Pending/InProgress/Completed/Failed）。
- 分页读取增量数据并写入暂存。
- 从暂存执行幂等合并（Insert/Update/Skip）。
- 执行删除识别与删除应用。
- 先写变更日志/删除日志，再提交检查点。

### 2.3 配置与治理层
- StartTimeLocal 按本地时间解析并禁止 `Z`/offset。
- ExcludedColumns 冲突校验（UniqueKeys/Cursor/软删除列）。
- 删除对比分段与并行参数可配置。
- 保留期治理具备总开关、dry-run 与审计基础能力。

## 3. 代码缺陷与逻辑 BUG 分析

以下为基于当前代码可直接确认的问题：

### 缺陷 1：同步失败时可能丢失原始异常，影响排障准确性
- 位置：`EverydayChain.Hub.Application/Services/SyncExecutionService.cs`
- 关键方法：`ExecuteBatchAsync`（异常处理分支，约 218-250 行）
- 现象：在主异常捕获后，会使用 3 秒超时令牌写入失败检查点；若检查点写入超时/取消，抛出的新异常可能覆盖原始业务异常。
- 风险：最终错误栈可能变成检查点写入超时，而不是最初的同步失败原因，增加定位难度。

### 缺陷 2：检查点读取失败被吞掉并回退为空字典，存在“误全量补偿”风险
- 位置：`EverydayChain.Hub.Infrastructure/Repositories/SyncCheckpointRepository.cs`
- 关键方法：`LoadAllAsync`（异常降级分支，约 98-109 行）
- 现象：`LoadAllAsync` 捕获异常后仅记录日志并返回空字典。
- 风险：当检查点文件损坏或读取异常时, 调用方会拿到“无检查点”结果, 窗口起点可能回退到 `StartTimeLocal`, 导致重复同步量骤增。

### 缺陷 3：目标表业务键字典大小写比较不统一，可能产生重复键/删除不一致
- 位置：`EverydayChain.Hub.Infrastructure/Repositories/SyncUpsertRepository.cs`
- 关键方法：`MergeFromStagingAsync`（目标表初始化分支，约 25 行）
- 现象：`_targetTables` 外层按表名忽略大小写，但内层 `ConcurrentDictionary<string, ...>` 未显式指定 `StringComparer.OrdinalIgnoreCase`。
- 风险：业务键若仅大小写不同，可能被视为不同记录；而删除识别侧大量使用忽略大小写比较，导致行为不一致。

### 缺陷 4：DryRun 删除仍写入 Delete 变更日志，语义容易误导
- 位置：`EverydayChain.Hub.Application/Services/DeletionExecutionService.cs`
- 关键方法：`ExecuteDeletionAsync`（生成 changeLogs 分支，约 71-81 行）
- 现象：即使 `DeletionDryRun=true`, 仍会为候选项写入 `SyncChangeLog`（`OperationType=Delete`）, 仅在 `SyncDeletionLog.Executed` 中区分是否实际执行。
- 风险：下游若只消费变更日志而未联表删除日志, 可能将“演练删除”误判为“真实删除”。

### 缺陷 5：删除候选可能重复，审计记录存在重复噪音
- 位置：`EverydayChain.Hub.Infrastructure/Repositories/SyncDeletionRepository.cs` + `EverydayChain.Hub.Application/Services/DeletionExecutionService.cs`
- 关键方法：`DetectDeletedKeysAsync`（候选生成）、`ExecuteDeletionAsync`（日志生成）
- 现象：删除应用前对业务键去重，但候选日志生成阶段未统一去重。
- 风险：在目标侧存在重复业务键数据异常时，会产生重复删除审计与重复删除变更日志，影响审计可读性与统计准确性。

### 缺陷优先级分级（用于排期）
- **High（优先修复）**：缺陷 1、缺陷 2、缺陷 4  
  判定依据：会直接影响故障定位准确性、同步边界稳定性或业务语义正确性，且可能在日常运行中高频触发。
- **Medium（次级修复）**：缺陷 3、缺陷 5  
  判定依据：通常在特定数据条件下触发（如键大小写混用、目标数据重复异常），影响一致性与审计质量，但对主链路可用性影响相对较低。

## 4. 当前明显短板（非 BUG，但影响工程质量）

1. **缺少独立测试项目**：当前解决方案没有专门单元测试/集成测试工程，回归验证主要依赖构建和运行态验证。  
2. **Oracle 读取与目标写入仍以基础实现为主**：当前读取器与仓储层仍以内存/基础实现为主，距离真实生产链路还需要进一步落地。  
3. **指标目前主要以日志输出**：尚未形成统一监控指标管道（如集中时序指标平台接入）。

## 5. 当前验收方式（现状说明）

受当前工程结构影响（未拆分独立测试项目），当前迭代主要采用如下验收方式：

- **构建验收**：执行 `dotnet build EverydayChain.Hub.sln`，确保项目可编译通过。  
- **基础测试验收**：执行 `dotnet test EverydayChain.Hub.sln`（当前无独立测试工程，主要用于验证解决方案测试入口无阻塞）。  
- **文档一致性验收**：新增/删除文件后同步更新 `README.md` 的文件树、文件作用说明与本次更新内容。

---

## 6. 结论

当前程序已具备“可运行的同步骨架能力”，尤其在编排、窗口、幂等、删除治理、检查点和审计方面结构完整；  
但在异常保真、检查点异常降级策略、业务键一致性与 dry-run 日志语义方面存在可确认的实现缺陷；  
建议按严重程度优先修复 High 项（缺陷 1/2/4），再处理 Medium 项（缺陷 3/5），以在最短路径内降低误判与运维排障成本。
