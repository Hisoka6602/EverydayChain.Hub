# 逐文件代码检查方案

## 1. 目标与边界
- 目标：逐文件排查影分身代码、过度设计、冗余代码、性能问题、逻辑问题、并发竞态问题、以及违反 `.github/copilot-instructions.md` 的实现，确保无遗漏。
- 边界：本轮仅执行检查与记录，不改业务代码；修复动作在后续分批 PR 落地。
- 交付：输出逐文件检查记录、问题清单、分批修复计划与每批次验收结论。

## 2. 待确认项（未确认前不默认实现）
- 是否将 `scripts/*.sh` 纳入与 C# 同等级的逐文件规则审查（当前建议纳入基础安全与可维护性检查）。
- 文档类文件（`*.md`）是否需要纳入“规则一致性”检查（当前建议仅检查与 README/治理规则联动一致性）。
- 本次分批 PR 的优先级是否固定为：`P0 可靠性/竞态` → `P1 逻辑/性能` → `P2 冗余/结构`。

## 3. 全量文件盘点与分组
1. 生成文件清单（排除 `bin/obj/.git`）并冻结为本轮基线。
2. 按目录分组检查，至少覆盖：
   - `.github`（治理规则与 CI 校验）
   - `EverydayChain.Hub.Domain`
   - `EverydayChain.Hub.Application`
   - `EverydayChain.Hub.Infrastructure`
   - `EverydayChain.Hub.Host`
   - `EverydayChain.Hub.SharedKernel`
   - `EverydayChain.Hub.Tests`
   - 根目录 `*.md` 与 `scripts`
3. 每个分组必须有“已检查文件计数 / 总文件计数”，防止遗漏。

## 4. 单文件检查维度（逐项打勾）
每个文件按以下维度逐项检查并记录“通过/问题/备注”：

1. 影分身与冗余
   - 是否存在同职责重复实现、复制粘贴分叉、无差异近似代码块。
   - 是否存在仅一层透传且无语义增益的方法。
2. 设计合理性
   - 是否出现过度抽象、过度分层、命名与职责不一致。
   - 是否违反 DDD 分层与接口归属规则。
3. 逻辑正确性
   - 条件分支、边界值、空值路径、异常路径是否完整。
   - 配置默认值与注释描述是否一致。
4. 并发与竞态
   - 共享状态读写是否具备线程安全保障。
   - 后台任务、批处理、重试与取消路径是否存在竞态窗口。
5. 性能与资源
   - 热路径是否有重复遍历、循环内重复分配、N+1 查询。
   - I/O、数据库、日志调用是否在高频路径造成不必要开销。
6. 规则合规（重点对照 `.github/copilot-instructions.md`）
   - 时间语义是否严格本地时间（禁止 UTC API）。
   - 命名空间与目录、类文件拆分、枚举/配置/事件放置是否合规。
   - 日志与异常、配置注释、README 联动、危险动作门禁是否合规。

## 5. 执行步骤（逐文件无遗漏流程）
1. **建账**：创建“文件台账 + 检查状态”表，推荐使用 Markdown 表格或 CSV，并固定机器可识别状态枚举值：`未检查|进行中|已完成`。台账字段建议如下：
   - 文件路径
   - 目录分组
   - 检查人
   - 检查日期
   - 检查状态（仅允许 `未检查|进行中|已完成`）
   - 问题编号列表
   - 复核人
   - 复核结论
   
   示例（Markdown）：

   | 文件路径 | 目录分组 | 检查人 | 检查日期 | 检查状态 | 问题编号列表 | 复核人 | 复核结论 |
   | --- | --- | --- | --- | --- | --- | --- | --- |
   | EverydayChain.Hub.Application/Services/SyncExecutionService.cs | Application | 张三 | 2026-04-09 | 已完成 | P0-001,P1-003 | 李四 | 通过 |
   | EverydayChain.Hub.Infrastructure/Repositories/SyncTaskConfigRepository.cs | Infrastructure | 张三 | 2026-04-09 | 进行中 | P1-008 | 李四 | 待复核 |
2. **逐组推进**：按目录分组顺序检查，单次只处理一个分组，避免跨组遗漏。
3. **逐文件打卡**：每检查完一个文件，立即更新状态与问题编号。
4. **组内复核**：每个分组结束执行二次复核，确认“已完成数 = 总数”。
5. **全量对账**：汇总所有分组，确认无“未检查”状态文件。

## 6. 问题分级与分批 PR 策略
- `P0`：并发竞态、数据错误、严重逻辑缺陷、违反硬性治理规则（优先单独 PR）。
- `P1`：性能热点、结构性冗余、可导致后续缺陷扩散的问题。
- `P2`：命名/注释/轻度重复等低风险治理项。

分批原则：
1. 单个 PR 聚焦同一主题与同一层级，避免混杂。
2. 每个 PR 必须包含：问题来源文件、影响范围、修复策略、验收证据、剩余待办。
3. 未确认项不进入实现 PR，先在 PR 描述或 Issue 完成确认闭环。

- 是否进入修复 PR：是/否（计划批次）

- [ ] 全量文件已建账并完成逐文件状态标记
- [ ] 每个目录分组均完成“已检查数=总数”复核
- [ ] 已输出按 P0/P1/P2 分级的问题清单
- [ ] 已形成可执行的分批 PR 列表（含每批目标文件范围）
- [ ] 已记录全部待确认项并完成责任人指派
