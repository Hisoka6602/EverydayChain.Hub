# EverydayChain.Hub 年度维护清单

## 说明

本清单适用于 EverydayChain.Hub 同步服务的**月度、季度与年度**例行巡检，  
确保服务在长期无人值守场景下保持稳定运行。  
每次巡检结束后，应记录日期、操作人及巡检结论（正常/异常/已处理）。

---

## 月度巡检（每月执行一次）

### M1. 磁盘与文件健康

- [ ] 执行 `bash scripts/health-check.sh` 一键体检，确认全部 PASS
- [ ] 检查 `data/` 目录下 `.json` 文件体积，单文件超过 50 MB 时评估是否需要缩减同步表范围或启用归档
- [ ] 检查 `data/` 目录下压缩归档 `.json.gz` 数量，超过 `TargetStoreArchiveMaxCount` 时确认自动清理已生效
- [ ] 检查 `logs/` 目录总大小，超过 5 GB 时手动归档并清理 60 天以前的日志文件

### M2. 日志质量审查

- [ ] 统计上月 `[ERROR]` 日志行数，分析高频错误类型
- [ ] 检查是否出现"压缩归档旧版本目标快照文件失败"等归档异常告警
- [ ] 检查是否出现"单表内存水位告警"，评估是否需要调整 `TableMemoryWarningThresholdMb`
- [ ] 检查是否出现"本轮同步存在失败表"，统计失败表与失败率趋势
- [ ] 检查是否出现"同步整轮执行超时"，评估是否需要调整 `TableSyncTimeoutSeconds`

### M3. 数据一致性抽查

- [ ] 随机抽取 1~2 张同步表，核对源端与目标端记录数是否一致（允许 lag 范围内的差异）
- [ ] 检查检查点文件 `sync-checkpoints.json`，确认最近游标时间在合理范围（不超过 2 × `DefaultMaxLagMinutes`）
- [ ] 确认无大量"跳过行数（Skip）"堆积，避免静默数据漂移

### M4. 配置项审核

- [ ] 确认 `appsettings.json` 中 Oracle 连接串、SQL Server 连接串未过期
- [ ] 确认 `SyncJob.Tables` 中各表 `Enabled` 状态与业务需求一致
- [ ] 确认危险动作开关 `RetentionJob.AllowDangerousAction`、`Delete.DryRun` 等状态符合预期

## 季度巡检（每季度执行一次，涵盖全部月度项）

### Q1. 弹性策略评估

- [ ] 查看 DangerZoneExecutor 熔断触发频率日志，评估 `CircuitBreakerFailureRatio`、`CircuitBreakerBreakDurationSeconds` 是否仍然合适
- [ ] 评估 OracleSourceReader 重试次数是否频繁，考虑是否需要调整 `MaxRetryAttempts`、`RetryBaseDelaySeconds`
- [ ] 评估 `TableSyncTimeoutSeconds` 是否需要根据大表实际耗时上调

### Q2. 内存与磁盘水位复盘

- [ ] 查看每月最高内存水位告警记录，评估是否需要下调 `TableMemoryWarningThresholdMb`
- [ ] 查看空闲驱逐触发频率，评估 `IdleEvictionThresholdMinutes` 是否合理
- [ ] 评估 `TargetStoreArchiveMaxCount` 设置与实际磁盘占用的匹配情况

### Q3. 日志归档与清理

- [ ] 将 90 天以前的日志文件压缩归档到冷存储（如 `/archive/logs/`）
- [ ] 确认 nlog.config `maxArchiveFiles` 配置下的归档文件已按预期轮转，无异常残留

### Q4. 数据保留期治理

- [ ] 检查 `RetentionJob` 执行记录，确认各分表保留期清理符合 `KeepMonths` 配置
- [ ] 若 `RetentionJob.AllowDangerousAction=false`，评估是否需要开放一次性清理并在执行后及时关闭

### Q5. 依赖组件版本检查

- [ ] 检查 Oracle 驱动（Oracle.ManagedDataAccess.Core）是否有安全补丁需要升级
- [ ] 检查 .NET 运行时补丁版本，评估是否需要滚动升级（维护窗口内）
- [ ] 检查 NLog、Polly、Entity Framework Core 等关键依赖是否有重大安全漏洞通告

## 年度巡检（每年执行一次，涵盖全部月度与季度项）

### Y1. 灾难恢复演练

- [ ] 模拟检查点文件损坏场景（备份 → 清空 → 重启），验证服务能否从头自恢复同步
- [ ] 模拟目标快照文件损坏场景（备份 → 清空 → 重启），验证服务能否从压缩归档恢复
- [ ] 演练 Oracle 连接断开 30 分钟后的熔断恢复全流程，验证依赖恢复后自动回到正常吞吐
- [ ] 演练 SQL Server 连接断开场景，验证重试策略与日志输出符合预期

### Y2. 全量数据一致性验证

- [ ] 对全部启用同步的表执行一次全量行数比对（源端 vs 目标端）
- [ ] 检查各表最大游标与业务期望进度是否一致，排查是否存在长期未推进的"僵尸表"

### Y3. 安全审计

- [ ] 确认 Oracle 连接串、SQL Server 连接串中无明文密码残留（应通过环境变量或密钥服务注入）
- [ ] 确认日志文件中无敏感业务字段（如工号、身份证等）明文输出
- [ ] 审计 `DangerZone` 相关危险操作执行记录，确认所有批量删除/更新均有对应的审计日志

### Y4. 容量规划

- [ ] 统计过去一年各同步表数据增速，预测未来 12 个月磁盘与内存需求
- [ ] 评估是否需要扩容磁盘、调整 `StartupMinFreeSpaceMb` / `WriteMinFreeSpaceMb` 阈值
- [ ] 评估是否需要增加 `MaxParallelTables` 以提升同步吞吐

### Y5. 文档与手册更新

- [ ] 更新 README.md 文件树与各层说明，确保与当前代码一致
- [ ] 更新本维护清单，删除已废弃的检查项，补充新增的检查项
- [ ] 更新 Oracle到SQLServer同步实施计划.md，核对各条目实现状态
- [ ] 更新值班处置手册（若有），确保新同学可以独立处理常见告警

## 快速参考：常见异常处理

| 告警关键字 | 可能原因 | 建议处理 |
|---|---|---|
| `磁盘可用空间不足` | 日志/归档积压 | 清理旧日志归档，调低 `TargetStoreArchiveMaxCount` |
| `单表内存水位告警` | 大表内存膨胀 | 检查 `IdleEvictionThresholdMinutes` 是否过大 |
| `同步整轮执行超时` | 大表同步耗时超限 | 上调 `TableSyncTimeoutSeconds`，排查 Oracle 慢查询 |
| `熔断器打开` | Oracle/SQL Server 连续失败 | 检查网络/连接串，等待依赖恢复后自动关闭 |
| `压缩归档旧版本目标快照文件失败` | 磁盘空间不足或权限问题 | 清理磁盘后，删除残留 .bak 文件，重启服务 |
| `读取同步目标落地文件失败` | 文件损坏或格式非法 | 备份后清空该 .json 文件，重启服务重新同步 |
