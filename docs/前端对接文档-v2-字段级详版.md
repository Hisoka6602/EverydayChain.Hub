# 前端对接文档（v2 字段级详版）

## 1. 文档目标

- 对接页面：`https://wkrdemo.zeyekeji.com/v2`
- 对接项目：`EverydayChain.Hub`
- 编写日期：`2026-07-10`
- 使用对象：前端程序员、联调人员、后端协作人员

这份文档的目标不是只告诉前端“有哪些接口”，而是把下面 4 件事一次性说清楚：

1. 每个页面对应哪些接口
2. 页面上的每个区块、按钮、表格列分别用哪个接口、哪个字段
3. 哪些页面要拼多个接口才能渲染完整
4. 哪些字段或按钮当前后端还没完全支持，前端应该怎么降级

## 2. 重要说明

### 2.1 这次对接的依据

本次文档基于两部分信息整理：

1. 当前仓库中的后端控制器、请求契约、响应契约
2. `v2` 页面实际展示的 UI 结构与字段

### 2.2 一个关键事实

`v2` 页面当前还是静态演示页，不是已经真实接了接口的前端页面。

也就是说：

1. 页面上现在看到的数字、表格、按钮，大多还是 mock 演示数据
2. 前端 bundle 中没有真实的 `fetch / axios / request` 调用链
3. 所以这份文档采用的是“按页面 UI 需求反推真实 API 对接方案”

### 2.3 当前总体结论

当前后端已经覆盖了大部分 `v2` 页面需要的查询、导出、波次清理能力，前端可以开始联调。

但仍有以下缺口：

1. 没有单独的“读码平台识别率”字段
2. 没有“当前波次体积 / 当前波次重量”字段
3. 波次状态只有粗粒度状态，和演示页更细的状态不完全一致
4. 导出页的 `updatedAt` 不是导出项真实最近同步时间
5. 演示页中的 `导入` / `导入列表` 当前没有对应后端接口

## 3. 全局对接约定

## 3.1 Base URL

- 本地默认：`http://localhost:5188`
- Swagger：`http://localhost:5188/swagger`

## 3.2 请求方式

- 除下载接口外，所有查询/操作接口统一按 `POST + application/json`
- 不建议前端走 querystring 模式

## 3.3 时间字段

所有时间字段都是“本地时间语义”，前端必须传本地时间字符串，不要附带 `Z`。

推荐格式：

```json
{
  "startTimeLocal": "2026-07-10T00:00:00",
  "endTimeLocal": "2026-07-10T23:59:59"
}
```

## 3.4 JSON 命名

前端按 `camelCase` 收字段。

示例：

- `StartTimeLocal` -> `startTimeLocal`
- `WaveCode` -> `waveCode`
- `IsSuccess` -> `isSuccess`

## 3.5 统一响应格式

除文件流接口外，统一使用：

```ts
type ApiResponse<T> = {
  isSuccess: boolean;
  message: string;
  data: T | null;
};
```

### 成功示例

```json
{
  "isSuccess": true,
  "message": "Query succeeded.",
  "data": {}
}
```

### 失败示例

```json
{
  "isSuccess": false,
  "message": "WaveCode cannot be empty.",
  "data": null
}
```

## 3.6 下载接口格式

以下接口返回文件流，不是 `ApiResponse<T>`：

- CSV：`text/csv; charset=utf-8`
- Excel：`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`

前端统一按 `blob` 处理。

## 3.7 鉴权和特殊请求头

当前代码里未看到控制器级显式鉴权要求。

前端建议：

1. 普通查询先按无特殊业务头调用
2. `wave-cleanup/execute` 调用时，建议带：

```http
X-Operator-Id: 当前登录人账号或工号
```

## 5. 公共筛选栏

页面顶部的公共筛选栏包含：

1. 时间来源
2. 时间范围

### 5.1 时间来源

演示页显示：

- `WMS时间`
- `DWS时间`（禁用）

当前后端实际情况：

1. 所有接口都没有 `timeSource` 字段
2. 也没有支持 `DWS时间` 的分支逻辑
3. 所以前端实际请求时不需要发送 `timeSource`

前端建议：

1. `WMS时间` 固定选中
2. `DWS时间` 继续禁用
3. 后续如果后端补充多时间源能力，再增加字段

### 5.2 时间范围

除个别默认可空接口外，绝大多数查询都要求：

- `startTimeLocal`
- `endTimeLocal`

前端建议将顶部时间范围作为整个页面的全局查询条件。

## 6. 汇总看板

## 6.1 页面结构

`汇总看板` 这个页签，页面上实际包含 4 组数据区块：

1. 左上：拆零 / 整件未分拣数量
2. 中上：运行概览
3. 右上：实时分拣波次 + 识别率 + 下载/回传进度
4. 下方：核心任务进度统计

这 4 组并不是一个接口就能全部返回，前端必须拼接口。

## 6.2 推荐请求顺序

当页面首次进入、时间范围变化、用户手工刷新时，建议按以下顺序：

1. `POST /api/v1/dashboard/overview`
2. `POST /api/v1/waves/current`
3. 如果第 2 步有 `waveCode`，则调用 `POST /api/v1/waves/summary`
4. 再调用 `POST /api/v1/waves/list`，从波次列表中找到当前波次那一行

### 6.2.1 原因

- `dashboard/overview` 负责全局统计
- `waves/current` 负责当前波次
- `waves/summary` 负责当前波次的核心进度
- `waves/list` 负责补充当前波次拆零/整件数量与比例

### 前端使用方式

不是整表直接全量展示到汇总看板，而是：

1. 先拿到 `currentWave.waveCode`
2. 再从 `waves/list` 返回的 `items` 里找到 `waveId === currentWave.waveCode` 的那一行
3. 用这一行补充当前波次的拆零/整件统计

### 这个接口负责页面哪些字段

| 页面文案 | 取值字段 |

| 拆零总数 | `splitTotal` |
| 整件总数 | `fullTotal` |

- 体积
- 重量

1. 隐藏
2. 或使用全局总量，但文案必须改成“统计范围总体积 / 统计范围总重量”

## 15. 当前缺口与前端降级策略

## 15.1 缺口 1：读码平台识别率

现状：

- 页面需要两个识别率
- 后端只有一个 `recognitionRatePercent`

前端建议：

1. 隐藏“读码平台识别率”
2. 或用同一个值顶住，但界面上要明确是临时方案

## 15.2 缺口 2：单波次体积 / 重量

现状：

- 页面展示更像当前波次数据
- 后端只有统计范围总量

前端建议：

1. 隐藏
2. 或文案改成“统计范围总体积 / 总重量”

## 15.3 缺口 3：波次状态颗粒度不足

现状：

- 页面 mock 更细
- 后端真实状态更粗

前端建议：

1. 先按后端值映射展示
2. 不要前端自行拆更多状态

## 15.4 缺口 4：导入按钮无接口

前端建议：

1. 隐藏
2. 置灰
3. 点击提示“待开放”

## 15.5 缺口 5：导出目录的 `updatedAt`

现状：

- 当前更接近“目录生成时间”
- 不是导出项真实最近同步时间

前端建议：

1. 若 UI 必须准确，文案改成“记录时间”
2. 若仍显示“最后同步”，需要接受口径不完全准确

## 16. 推荐前端开发顺序

建议联调顺序：

1. 先打通顶部公共时间范围 + 汇总看板
2. 再打通波次明细
3. 再打通进度详情
4. 再打通回流筛选
5. 再打通箱子追踪
6. 最后联调波次清理和全部导出

这样可以优先完成主查询链路，再补充操作链路和下载链路。
