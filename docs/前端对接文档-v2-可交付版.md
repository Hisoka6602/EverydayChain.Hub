# 前端对接文档（v2 页面可交付版）

## 1. 文档说明

- 对接目标页面：`https://wkrdemo.zeyekeji.com/v2`
- 对接后端项目：`EverydayChain.Hub`
- 文档日期：`2026-07-10`
- 文档用途：交付给前端程序员，作为当前后端接口的正式对接依据

本文件基于当前仓库内控制器、请求契约、响应契约和页面 `v2` 的实际展示内容整理。

需要特别说明：

1. `v2` 前端页面目前仍是静态演示页，页面 bundle 中没有实际接口调用代码。
2. 因此本文档采用“按页面功能需求反推接口”的方式编写。
3. 当前后端已经覆盖大部分页面查询、导出和清理能力，但仍有少量字段和交互差异，已在本文档中单独标出。

## 2. 总体结论

当前前端可以开始对接，建议按本文档先完成以下页面：

- 汇总看板
- 波次明细
- 进度详情
- 回流筛选
- 箱子追踪
- 波次清理
- 导出明细

当前已知未完全闭环的点：

1. 总看板只有一个 `recognitionRatePercent`，没有单独的“读码平台识别率”字段。
2. 当前接口没有“按单个波次返回体积、重量”的字段。
3. 当前波次状态语义只有 `Sorting`、`Completed`、`ExceptionPending` 三档，和演示页中的 `待启动 / 待重下发 / 预排异常 / 分拣中 / 已完成` 不完全一致。
4. 导出中心 `updatedAt` 目前是目录生成时间，不是每个导出项的真实最近同步时间。
5. 导出中心返回的 `endpoint` 仅提供接口路径，不提供额外入参说明；其中部分导出接口还需要前端额外补传 `waveCode`。
6. 演示页中的 `导入`、`导入列表` 按钮，当前后端没有对应接口。

前端建议做法不是等待全部补齐，而是：

1. 先按当前接口完成主链路。
2. 对缺口字段做兼容显示。
3. 对暂未落地的按钮先隐藏、置灰或保留纯前端提示。

## 3. 基础约定

### 3.1 Base URL

- 本地默认：`http://localhost:5188`
- Swagger：`http://localhost:5188/swagger`
- 生产/测试环境以部署配置为准

### 3.2 请求方式

- 查询类接口几乎全部使用 `POST`
- 请求体统一使用 `application/json`
- 前端统一按 `POST + JSON Body` 对接，不建议再走 querystring 调试模式

### 3.3 时间字段约定

- 所有时间字段均为“本地时间语义”
- 不要传 UTC `Z`
- 推荐格式：`2026-07-10T08:00:00`

前端统一约定：

```json
{
  "startTimeLocal": "2026-07-10T00:00:00",
  "endTimeLocal": "2026-07-10T23:59:59"
}
```

### 3.4 JSON 命名风格

当前代码未看到自定义 JSON 命名策略，按 ASP.NET Core 默认 Web 序列化行为，前端应按 `camelCase` 接收字段。

例如后端 C#：

- `IsSuccess` -> JSON `isSuccess`
- `StartTimeLocal` -> JSON `startTimeLocal`
- `WaveCode` -> JSON `waveCode`

### 3.5 统一响应结构

除文件下载接口外，统一响应结构如下：

```ts
type ApiResponse<T> = {
  isSuccess: boolean;
  message: string;
  data: T | null;
};
```

成功响应示例：

```json
{
  "isSuccess": true,
  "message": "Wave list query succeeded.",
  "data": {
    "startTimeLocal": "2026-07-10T00:00:00",
    "endTimeLocal": "2026-07-10T23:59:59",
    "items": []
  }
}
```

失败响应示例：

```json
{
  "isSuccess": false,
  "message": "WaveCode cannot be empty.",
  "data": null
}
```

### 3.6 文件下载接口约定

以下接口返回文件流，不走 `ApiResponse<T>` 包装：

- CSV：`text/csv; charset=utf-8`
- Excel：`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`

前端下载时请使用：

- `responseType: 'blob'`
- 从响应头 `Content-Disposition` 中读取文件名

### 3.7 鉴权与请求头

当前控制器代码中未看到控制器级别的显式鉴权声明。

前端请按以下规则处理：

1. 常规查询接口先不额外带业务头。
2. 波次清理正式执行接口建议带上操作人：

```http
X-Operator-Id: 当前登录用户名或工号
```

3. 真实环境如果网关还有 token、cookie 或单点登录要求，以环境网关规则为准。

## 4.2 汇总看板

### 4.2.1 推荐接口编排

页面初始化或时间变化时，建议按以下顺序调用：

1. `POST /api/v1/dashboard/overview`
2. `POST /api/v1/waves/current`
3. 如果第 2 步返回了 `waveCode`，再调用 `POST /api/v1/waves/summary`
4. 如果前端还要拿当前波次的 `split/full` 数量与占比，再调用 `POST /api/v1/waves/list`，从返回列表里找到当前波次那一行

推荐原因：

1. `dashboard/overview` 负责全局概览
2. `waves/current` 负责“当前自动识别波次”
3. `waves/summary` 负责当前波次的核心进度指标
4. `waves/list` 负责补齐当前波次的拆零/整件数量和占比

3. 弹出确认框，展示 `identifiedCount`
4. 用户确认后调 `execute`

## 6. 前端调用建议

## 6.1 查询接口

统一封装为：

```ts
async function postJson<T>(url: string, body: unknown): Promise<T> {
  const res = await request<ApiResponse<T>>(url, {
    method: "POST",
    data: body,
  });

  if (!res.isSuccess || !res.data) {
    throw new Error(res.message || "接口调用失败");
  }

  return res.data;
}
```

## 6.2 下载接口

```ts
async function downloadFile(url: string, body: unknown) {
  const res = await request(url, {
    method: "POST",
    data: body,
    responseType: "blob",
    getResponse: true,
  });

  return res;
}
```

## 8. 推荐联调优先级

建议联调顺序：

1. 先通 `dashboard/overview`、`waves/current`、`waves/list`
2. 再通 `waves/options`、`waves/zones`
3. 再通 `recirculations/summary`、`box-tracking/query`
4. 最后联调 `wave-cleanup/*` 和全部下载接口

原因：

1. 这条顺序能先把所有列表和看板主链路跑通
2. 清理和导出都依赖前面的查询上下文

## 9. 联调时前端需要特别注意的错误场景

1. 时间范围非法会返回 `400 + ApiResponse.fail`
2. `waveCode` 为空时，`waves/summary`、`waves/details`、`waves/zones`、`wave-cleanup/*` 会失败
3. `pageNumber` 和 `pageSize` 超范围会返回 400
4. 下载接口失败时通常不会返回 blob，而会返回 JSON 错误体，前端要能兼容解析

## 10. 最终落地建议

如果目标是“先交付一个可上线联调版本”，前端应按下面策略执行：

1. 所有查询和导出链路按本文档现有接口直接落地
2. 对 `读码平台识别率`、`当前波次体积/重量`、`导入类按钮` 做兼容降级
3. 对 `waveCode` 依赖型导出项增加前端前置校验
4. 对状态文案统一走前端映射，不要写死在页面 JSX 内

如果后续后端补齐缺口，前端只需要替换字段映射，不需要重做页面结构。
