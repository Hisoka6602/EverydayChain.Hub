# 数据同步机制详解

本文档详细描述 EverydayChain.Hub 项目中 Oracle → SQL Server 数据同步链路的实现机制，涵盖触发调度、数据读取、变化感知（新增/修改/删除）、变更写入、检查点与批次状态流转等核心环节。

---

## 一、整体架构

系统采用四层架构，同步链路的核心类依赖关系如下：

```
SyncBackgroundWorker（入口：Host 层后台 Worker）
    ↓ 轮询调用
SyncOrchestrator（编排层：按优先级并发调度多表）
    ├→ ISyncTaskConfigRepository（读取表定义与同步参数）
    ├→ ISyncCheckpointRepository（读取上次成功游标）
    ├→ ISyncWindowCalculator（计算本次增量窗口）
    └→ ISyncExecutionService（执行单表批次同步）
            ├→ IOracleSourceReader（读取 Oracle 源端增量数据）
            ├→ ISyncStagingRepository（内存暂存分页数据）
            ├→ ISyncUpsertRepository（幂等合并：INSERT/UPDATE/SKIP）
            ├→ IDeletionExecutionService（删除同步）
            │       └→ ISyncDeletionRepository（删除检测与执行）
            ├→ ISyncBatchRepository（批次状态流转）
            ├→ ISyncChangeLogRepository（变更审计日志）
            ├→ ISyncDeletionLogRepository（删除审计日志）
            └→ ISyncCheckpointRepository（提交新游标检查点）
```

## 二、同步任务触发与调度

### 触发方式：后台轮询

**文件**：`EverydayChain.Hub.Host/Workers/SyncBackgroundWorker.cs`

`SyncBackgroundWorker` 继承 `BackgroundService`，随应用启动，进入 `while (!stoppingToken.IsCancellationRequested)` 轮询循环。每次循环调用一次 `SyncOrchestrator.RunAllEnabledTableSyncAsync()`，完成后等待 `PollingIntervalSeconds`（默认 60 秒）再进行下一轮。

### 多表调度策略

**文件**：`EverydayChain.Hub.Application/Services/SyncOrchestrator.cs`

```csharp
// 按优先级排序，高优先级（High）排在前面
var orderedDefinitions = definitions
    .OrderByDescending(x => x.Priority == SyncTablePriority.High)
    .ThenBy(x => x.TableCode, StringComparer.OrdinalIgnoreCase)
    .Select((definition, index) => (definition, index))
    .ToList();

// 通过 MaxParallelTables 限制同时执行的表数，避免源端过载
var maxParallelTables = await configRepository.GetMaxParallelTablesAsync(ct);
await Parallel.ForEachAsync(orderedDefinitions, new ParallelOptions
{
    MaxDegreeOfParallelism = maxParallelTables,
    CancellationToken = ct,
}, async (item, token) =>
{
    results[item.index] = await RunTableSyncAsync(item.definition.TableCode, token);
});
```

## 三、数据读取策略：增量游标窗口

### 3.1 同步窗口计算

**文件**：`EverydayChain.Hub.Application/Services/SyncWindowCalculator.cs`

```
WindowStart = 上次成功游标（LastSuccessCursorLocal）OR 配置的初始时间（StartTimeLocal）
WindowEnd   = 当前本地时间 - MaxLagMinutes（避免读到乱序延迟数据）
```

**示例**：
- 上次成功游标：`10:00:00`
- `MaxLagMinutes = 10`，当前时间：`10:30:00`
- **本次窗口**：`(10:00:00, 10:20:00]`

### 3.2 分页增量读取

**文件**：`EverydayChain.Hub.Infrastructure/Repositories/OracleSourceReader.cs`

- 按 `CursorColumn`（通常为最后修改时间列）在窗口 `(WindowStart, WindowEnd]` 范围内过滤数据
- 按游标列升序排列，再按业务键稳定排序，确保分页顺序确定性
- 每页最多读取 `PageSize` 条（默认 5000），全部读完为止
- 读取阶段自动过滤掉 `ExcludedColumns` 配置的列

## 四、如何感知数据源变化

### 4.1 新增与修改（Upsert）的识别

**文件**：`EverydayChain.Hub.Infrastructure/Repositories/SyncUpsertRepository.cs`，方法：`MergeFromStagingAsync`

每读取一页源端数据后，进入幂等合并逻辑：

```
对每行源端数据：
  1. 过滤 ExcludedColumns
  2. 按 UniqueKeys 拼接业务键（例：OrderNo="ORD001", LineNo=1 → "ORD001|1"）
  3. 查询内存目标仓（_targetTables[tableCode]）中是否存在该业务键：
     ├─ 不存在          → INSERT（写入目标仓，计 InsertCount++）
     ├─ 存在 + 数据相同  → SKIP（幂等跳过，计 SkipCount++）
     └─ 存在 + 数据不同  → UPDATE（覆盖目标仓，计 UpdateCount++）
  4. 每行处理完成后推进最大游标（MaxCursor）
```

**逐字段对比逻辑**（`AreRowsEqual` 方法）：
- 先比较字段数量，不同直接判不等
- 再逐字段 `Equals` 比较值

> **关键设计**：目标仓是内存字典 `ConcurrentDictionary<业务键, 行数据>`，持久化到 `data/{tableCode}.json` 文件（原子替换写入）。服务重启后，首次访问该表时自动从文件加载，保证跨重启存活性。

### 4.2 删除的识别

**文件**：
- `EverydayChain.Hub.Infrastructure/Repositories/SyncDeletionRepository.cs`，方法：`DetectDeletedKeysAsync`
- `EverydayChain.Hub.Application/Services/DeletionExecutionService.cs`，方法：`ExecuteDeletionAsync`

删除感知采用**窗口内源端键集差集**策略，不做全量扫描：

```
Step 1：从 Oracle 源端读取本次窗口内所有行的业务键集合 → sourceKeys
Step 2：从内存目标仓读取所有行 → targetRows
Step 3：对 targetRows 中游标列在本窗口内的行进行遍历：
          若该行的业务键 NOT IN sourceKeys → 标记为删除候选（DeletionCandidate）
Step 4（可配置分段并行）：targetRows 按 CompareSegmentSize 分段，
          每段通过 Parallel.ForEachAsync 并行比对
```

**为什么只检查窗口内目标行？**
- 避免将历史数据（不在本次窗口内）误判为删除
- 只有游标列时间落在当前窗口内的目标行，才可能是本轮源端删除的数据

**删除策略**（由 `DeletionPolicy` 控制）：

| 策略 | 行为 |
|------|------|
| `SoftDelete` | 目标行追加 `IsDeleted=true`、`DeletedTimeLocal=当前时间` |
| `HardDelete` | 从内存字典中移除该业务键 |
| `Disabled` | 不执行删除 |

**DryRun 模式**：检测照常运行，写入删除候选审计日志（`SyncDeletionLog.Executed=false`），但**不写入变更日志**（`SyncChangeLog`），也**不实际执行删除**。

## 五、变更写入目标端的完整流程

每次批次执行遵循如下顺序（`SyncExecutionService.ExecuteBatchAsync`）：

```
[批次创建] → Pending
    ↓
[标记执行中] → InProgress
    ↓
[分页循环]
    ├─ 步骤1：OracleSourceReader.ReadIncrementalPageAsync()
    │         按游标窗口分页读取源端增量数据
    │
    ├─ 步骤2：SyncStagingRepository.BulkInsertAsync()
    │         将本页数据写入内存暂存区（BatchId + PageNo 索引）
    │
    ├─ 步骤3：SyncUpsertRepository.MergeFromStagingAsync()
    │         从暂存区取出数据，执行幂等合并（INSERT/UPDATE/SKIP）
    │         ✦ 仅有变更时写入持久化文件（原子替换）
    │
    ├─ 步骤4：SyncStagingRepository.ClearPageAsync()
    │         清理本页暂存数据（无论成功还是合并异常）
    │
    └─ 若 readResult.Rows.Count < PageSize → 退出循环（本窗口数据已读完）
    ↓
[步骤5：删除同步]
    DeletionExecutionService.ExecuteDeletionAsync()
    ├─ 检测删除候选（DetectDeletedKeysAsync）
    └─ 执行删除（ApplyDeletionAsync）→ SoftDelete 或 HardDelete
    ↓
[步骤6：先落变更日志与删除日志]（日志先写后提检查点原子边界）
    ├─ SyncChangeLogRepository.WriteChangesAsync()  （INSERT/UPDATE/DELETE 审计）
    └─ SyncDeletionLogRepository.WriteDeletionsAsync()（删除候选审计）
    ↓
[步骤7：提交检查点]（仅在日志写入成功后才提交）
    SyncCheckpointRepository.SaveAsync()
    └─ LastSuccessCursorLocal = 本批次最大游标
    ↓
[批次完成] → Completed，输出 Lag/Backlog/Throughput 指标日志
```

**异常路径**：
```
catch(Exception)
    ├─ 批次标记 Failed（保留原始异常，不覆盖）
    └─ 写入失败检查点（LastSuccessCursorLocal 保持不变 + LastError = ex.Message）
       ✦ 下次轮询从上次成功游标续跑，不丢失进度
```

## 六、检查点（Checkpoint）机制

**文件**：`EverydayChain.Hub.Infrastructure/Repositories/SyncCheckpointRepository.cs`

### 数据结构

```csharp
public class SyncCheckpoint
{
    public string TableCode { get; set; }            // 表编码
    public DateTime? LastSuccessCursorLocal { get; set; } // 上次成功游标（窗口起点依据）
    public string? LastBatchId { get; set; }          // 上次批次 ID
    public DateTime? LastSuccessTimeLocal { get; set; }  // 上次成功时间
    public string? LastError { get; set; }            // 最近错误信息（非空代表上次失败）
}
```

### 存储方式

- 存储在 JSON 文件中，路径由 `SyncJob.CheckpointFilePath` 配置（默认 `data/sync-checkpoints.json`）
- 使用 `SemaphoreSlim(1,1)` 文件锁保证并发安全
- 读取失败时**抛出异常**（而非静默降级为空），避免游标误回溯

### 检查点的核心作用

| 场景 | 行为 |
|------|------|
| 正常执行 | 记录最大游标，下次从此游标之后续读 |
| 任务失败 | 保留上次成功游标，只更新 LastError |
| 重启恢复 | 从文件读取，继续中断点 |
| 重试关联 | 检测到 LastError 非空时，关联上次失败批次 ID（ParentBatchId） |

## 七、批次（Batch）状态流转

**文件**：`EverydayChain.Hub.Infrastructure/Repositories/SyncBatchRepository.cs`

### 状态机

```
        ┌─────────────┐
        │  Pending(1)  │  ← CreateBatchAsync()
        └──────┬───────┘
               │ MarkInProgressAsync()
               ↓
        ┌──────────────────┐
        │  InProgress(2)   │
        └──────┬───────────┘
               │
      ┌────────┴─────────┐
      │（成功）           │（失败/取消）
      ↓                   ↓
  ┌──────────────┐  ┌──────────────┐
  │ Completed(3) │  │  Failed(4)   │
  └──────────────┘  └──────────────┘
```

- `Failed` 状态会同步更新 `_latestFailedBatchIndex`，编排层下次可通过 `GetLatestFailedBatchIdAsync` 获取并作为 `ParentBatchId` 传入，实现重试关联追溯。

## 八、完整调用链速览

```
应用启动 → SyncBackgroundWorker（后台 Worker）
    └─ 每 PollingIntervalSeconds 秒触发一次
        └─ SyncOrchestrator.RunAllEnabledTableSyncAsync()
            ├─ 读取全部启用表定义
            ├─ 按 Priority 排序（High 优先）
            └─ Parallel.ForEachAsync（MaxParallelTables 并发上限）
                └─ SyncOrchestrator.RunTableSyncAsync(tableCode)
                    ├─ 读取表定义、检查点、计算窗口
                    └─ SyncExecutionService.ExecuteBatchAsync(context)
                        ├─ CreateBatch / MarkInProgress
                        ├─ while(读 → 暂存 → 合并 → 清暂存)
                        ├─ ExecuteDeletion（检测删除 → 执行删除）
                        ├─ WriteChangeLogs + WriteDeletionLogs
                        ├─ SaveCheckpoint（新游标）
                        └─ CompleteBatch → 输出指标日志
```

## 九、关键设计特性总结

| 特性 | 实现机制 | 价值 |
|------|----------|------|
| **增量同步** | 游标列 + 滑动时间窗口 | 只读变更数据，避免全量扫描 |
| **幂等合并** | 业务键 + 逐字段对比 | 支持重跑不重复插入 |
| **删除感知** | 窗口内源键集差集 | 准确识别删除，不误删历史行 |
| **分页处理** | PageSize + 暂存区 | 大数据量下内存可控 |
| **崩溃恢复** | JSON 检查点文件 | 重启后从中断点续跑 |
| **原子写入** | 临时文件 + File.Replace | 防止落地文件损坏 |
| **审计完整** | 变更日志 + 删除日志 | 完整数据变更追溯 |
| **日志先写后提交** | 日志写入在检查点之前 | 避免漏审计的静默丢失 |
| **DryRun** | 检测执行，只写删除候选审计日志，不写变更日志，不执行删除 | 安全验证删除逻辑 |
| **并发控制** | 信号量 + ConcurrentDictionary | 多表并行同步线程安全 |
